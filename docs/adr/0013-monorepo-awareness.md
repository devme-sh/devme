# ADR-0013: Monorepo awareness and tooling integration

**Status**: Deferred (needs further grilling)
**Date**: 2026-05-25

## Context

Modern JS/TS projects increasingly use monorepo tooling (Turborepo, Vite+ `vp run`, Nx, Moon) to orchestrate builds and dev tasks across workspace packages. These tools handle task scheduling, caching, and dependency graphs between packages — but they don't manage infrastructure services (databases, caches, queues), don't provide unified log aggregation with filtering, and don't offer health-check-based startup ordering.

devme already solves these problems for single-project repos via `devme.toml` + the TUI. The question is how to extend this to monorepos where multiple packages each have their own dev server, and infra services are shared across them.

Additionally, devme's worktree model (multiple services with independent log streams shown in the TUI) maps naturally to monorepo packages — each package's dev process is a worktree with its own log pane.

We want to dogfood this with `devme-web` (the devme marketing site / future README editor), which will use TanStack Start + Vite+ and potentially a database.

## Decision

Defer to a future version. The high-level direction is:

### Detection via `devme init`

`devme init` should detect monorepo tooling and scaffold a `devme.toml` from the existing config:

- Detect `turbo.json`, `pnpm-workspace.yaml`, `vp` config, `nx.json`
- Enumerate workspace packages that have a `dev` script
- Offer each as a candidate service in the generated `devme.toml`
- Detect infra deps (docker-compose.yml, .env references to DATABASE_URL, etc.)

The result is explicit — devme doesn't silently discover services at runtime. `devme init` proposes, the user confirms, `devme.toml` is the source of truth.

### Orchestration model

devme sits above the monorepo tool, not beside it:

```
devme up
├── postgres (devme-managed, health-checked)
├── redis (devme-managed, health-checked)
├── app/web (delegates to `turbo dev --filter=web` or `vp dev`)
├── app/api (delegates to `turbo dev --filter=api` or `vp dev`)
└── packages/shared (watch mode via monorepo tool)
```

- Infra services: devme owns lifecycle, health checks, teardown
- App services: devme invokes the monorepo tool's dev command, captures stdout/stderr for the TUI
- Build tasks: stay in the monorepo tool entirely (devme doesn't replace `turbo build`)

### Log aggregation

Each workspace package's dev process gets its own log stream piped into the TUI. The TUI's existing worktree/pane model handles this — each package maps to a service with a named log stream, filterable and scrollable independently.

### Open questions (need grilling)

1. **Granularity**: Should each workspace package be a separate `[[service]]` in `devme.toml`, or should there be a `[monorepo]` section that auto-expands to services?
2. **Filtering**: Should `devme up` start all packages by default, or require explicit selection (like `turbo dev --filter`)?
3. **Dependency ordering**: Should devme parse the monorepo tool's dependency graph, or maintain its own `depends_on` in `devme.toml`?
4. **Shared vs per-package infra**: A monorepo might have one Postgres for all packages, or separate DBs per package. How does `devme.toml` express this?
5. **Nested devme.toml**: Should each workspace package optionally have its own `devme.toml` that the root one composes? Or flat?

## Consequences

### Positive

- devme becomes the single `devme up` entry point for monorepos, replacing the "run 4 terminals" problem
- Infra services (DBs, queues) are managed alongside app services with proper health checks and ordering
- The TUI provides better log visibility than monorepo tools' interleaved stdout
- Dogfooding on `devme-web` gives real signal before committing to the design
- Positions devme as complementary to (not competing with) monorepo tooling

### Negative

- Detection heuristics are fragile — monorepo configs vary widely
- Supporting multiple monorepo tools (Turbo, Vite+, Nx, Moon) is a maintenance surface
- Users may expect devme to fully replace their monorepo tool, causing confusion about responsibilities
- The `devme.toml` for a monorepo could become verbose

## Alternatives considered

### devme replaces the monorepo tool entirely

Rejected. Turborepo/Vite+ handle task caching, incremental builds, and package-level dependency graphs far better than devme should attempt. devme's value is orchestration and observability of running services, not build task scheduling.

### Automatic service discovery at runtime (no init step)

Rejected. Magic discovery leads to surprises — a package with a `dev` script isn't necessarily something you want running. Explicit `devme.toml` entries generated by `devme init` keep behavior predictable and diffable.

### One devme.toml per workspace package, no root composition

Rejected for the common case. Most monorepo dev workflows involve running multiple packages simultaneously with shared infra. A root-level `devme.toml` that composes services across packages matches how developers actually work. Per-package `devme.toml` could be supported later for standalone use.
