# ADR-0009: cargo-dist + three-path distribution

**Status**: Accepted
**Date**: 2026-05-23

## Context

devstack is a Rust CLI/TUI. It needs to reach Rust developers, macOS-default users, Linux developers, and CI environments. The "easiest install" is a moving target — for some users it's `cargo install`, for others `brew install`, for others `curl | sh`.

We could hand-roll a release pipeline (GitHub Actions workflow per platform, hand-written install.sh, manual Homebrew formula maintenance) or use `cargo-dist`, the de facto modern toolchain for Rust CLI distribution.

## Decision

Use `cargo-dist` to drive the release pipeline. Ship via three official install paths, in priority order:

1. **Curl-pipe-sh** — `curl -fsSL https://<host>/install | sh`. Detects OS + arch, downloads matching pre-built binary, installs to `~/.local/bin` (no sudo). Generated by cargo-dist with signature verification + checksum.
2. **Homebrew tap** — `brew install <org>/devstack/devstack`. Formula auto-generated and pushed to `homebrew-devstack` tap on every release. Apply for homebrew-core once we have a user base.
3. **`cargo install devstack`** — for Rust developers. Compiles from source.

Pre-built binaries on the GitHub Releases page for direct download (firewalled / auditable use cases). Targets: macOS arm64, macOS x86_64, Linux x86_64, Linux aarch64.

`devstack --version` reports both version and install method (curl-sh / brew / cargo / direct).

## Consequences

- One `dist.toml` config produces every install artifact + the GitHub Actions release workflow. Minimal maintenance overhead.
- Same approach as uv, ruff, polars, atuin — well-trodden path.
- macOS and Linux covered out of the box. Windows users get WSL (which uses the Linux binary).
- No self-update mechanism in v1 — users run the install command again to upgrade. `devstack upgrade` deferred to v1.1.

## Alternatives considered

**Hand-rolled release pipeline.** More control, more maintenance, more chances for cross-platform bugs. Out unless cargo-dist falls short of a specific need.

**Single install path (e.g., cargo only).** Excludes non-Rust users. The "is it easy to install?" question is the first thing potential users ask.

**Native Windows support in v1.** ratatui and tokio work on Windows, but the supervisor model (PTYs, Unix sockets) needs significant work to be cross-platform. WSL is the v1 path; native Windows is "maybe later, if anyone asks."
